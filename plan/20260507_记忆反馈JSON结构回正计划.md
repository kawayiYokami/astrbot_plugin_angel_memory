# 记忆反馈 JSON 结构回正计划

## 1. 背景与问题
- 当前实现一度偏离原始设计，把“新增记忆”和“合并来源”拆成了分裂结构，导致单条新记忆无法独立表达完整更新语义。
- 用户已确认最终协议应以 `memory_actions` 为中心：每个动作要么 `create`，要么 `merge`，而最终新记忆内容统一放在动作内部的 `memory` 字段中。
- 旧结构不仅提示词难理解，也容易让执行链路退化成“先建新记忆，再额外处理合并”，违背“merge 直接产出最终新记忆”的设计。

## 2. 目标
- 将反思 JSON 结构统一为 `feedback_data.memory_actions`。
- 让 `create` 与 `merge` 都以单个动作对象表达完整语义。
- 让 `merge` 直接生成最终新记忆，并替换 `source_memory_ids` 指向的旧记忆。
- 重写提示词，明确强调“合并相似记忆是非常重要的更新动作”，并给出专门示例。

## 3. 非目标
- 不修改记忆回想规则。
- 不修改召回策略、阈值、类型分组逻辑。
- 不保留旧 JSON 结构兼容层。
- 不改动笔记系统与研究子代理逻辑。

## 4. 目标 JSON 结构

```json
{
  "soul_state_code": "0110",
  "feedback_data": {
    "useful_memory_ids": ["id1", "id2"],
    "memory_actions": [
      {
        "action": "create",
        "memory": {
          "type": "knowledge",
          "judgment": "用户正在维护一个多阶段的记忆插件重构计划。",
          "reasoning": "本轮对话中用户明确描述了当前持续推进的开发任务。",
          "tags": ["记忆插件", "重构计划", "开发任务"]
        }
      },
      {
        "action": "merge",
        "source_memory_ids": ["31", "44"],
        "memory": {
          "type": "skill",
          "judgment": "用户偏好把复杂任务拆成可验证的小步骤执行。",
          "reasoning": "多轮对话都体现出用户偏好阶段性确认与逐步推进。",
          "tags": ["任务偏好", "执行方式"]
        }
      }
    ]
  }
}
```

说明：
- `memory_actions` 是唯一的记忆更新入口。
- `create` 只负责新增，不带 `source_memory_ids`。
- `merge` 必须同时带 `source_memory_ids` 和 `memory`。
- `merge` 的 `memory` 就是最终应保留的新记忆，不再额外依赖别处补一条“新记忆”。

## 5. 实施范围

### 5.1 提示词
- 文件：`llm_memory/prompts/memory_system_guide_async.md`
- 调整：
  - 输出格式改为 `memory_actions`
  - 删除旧的 `merge_groups` / 分桶 `new_memories`
  - 明确写出 `create` 与 `merge` 的约束
  - 单独增加“相似记忆合并”和“单条旧记忆修正替换”的示例
  - 明确强调：合并相似记忆对更新非常重要

### 5.2 反思解析与归一化
- 文件：`core/deepmind.py`
- 文件：`core/utils/memory_id_resolver.py`
- 调整：
  - 只解析 `feedback_data.memory_actions`
  - 校验动作类型仅为 `create` / `merge`
  - `merge` 透传并归一化 `source_memory_ids`
  - 继续支持短 ID 到完整 ID 的映射

### 5.3 反馈执行链
- 文件：`core/memory_runtime/protocol.py`
- 文件：`core/memory_runtime/simple_memory_runtime.py`
- 文件：`core/memory_runtime/vector_memory_runtime.py`
- 文件：`llm_memory/service/cognitive_service.py`
- 文件：`llm_memory/service/memory_manager.py`
- 文件：`llm_memory/components/memory_sql_manager.py`
- 调整：
  - 统一改用 `memory_actions`
  - `create` 直接新增
  - `merge` 直接用动作中的 `memory` 创建最终新记忆，并删除 `source_memory_ids` 旧记忆
  - 支持单源 `merge`，即“一条旧记忆被新记忆替换”

### 5.4 辅助展示与版本
- 文件：`core/utils/memory_injector.py`
- 文件：`metadata.yaml`
- 调整：
  - 注入侧展示按 `memory_actions` 提取的新记忆内容
  - 同步提升版本号，确保上游能识别接口语义变化

## 6. 验收标准
- 提示词与代码解析结构完全一致，均使用 `memory_actions`。
- 系统不再读取或执行旧的 `merge_groups` / 分桶 `new_memories` 结构。
- `merge` 可以处理多源合并，也可以处理单源替换。
- 合并后的最终保留内容由 `action.memory` 决定，而不是旧记忆模板。
- 运行日志和代码注释不再把本次协议写成旧结构。

## 7. 风险与注意事项
- 这是接口语义切换，提示词和执行链必须同次提交一起切换，不能半新半旧。
- 单源 `merge` 若继续沿用“复制 ID”伪造双源，会导致强度统计和删除逻辑异常，必须清理。
- 本次变更会影响线上行为，提交前必须同步更新 `metadata.yaml` 版本号。

## 8. 测试建议
- 构造纯新增场景，确认 `create` 正常落库。
- 构造两条相似旧记忆，确认 `merge` 后只保留动作里的最终新记忆。
- 构造单条旧记忆修正场景，确认单源 `merge` 能直接替换旧记忆。
- 构造错误动作，确认解析阶段会拒绝不合法结构。
