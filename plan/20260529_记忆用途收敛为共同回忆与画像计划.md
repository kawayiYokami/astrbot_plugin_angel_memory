# 记忆生成提示词重写：下放母项目方法论，确立「画像优先」

- 日期：2026-05-29
- 状态：待用户确认
- 负责模块：反思提示词（`llm_memory/prompts/sections/*.md`）

## 1. 背景与目标

母项目（大型 agent）已沉淀出一套成熟的记忆生成方法论（办事指南式、归属判断、
五问筛选、画像优先），效果优于本插件当前偏「记知识点」的提示词。

本次目标：**把母项目的方法论下放到本插件的反思提示词**，确立记忆生成的优先级——
**用户画像最高优先**，其次与用户的共同回忆/AI 自身经验，其它知识点照常生成但优先级最低。

关键纠偏（来自用户反复确认）：
- 这是**优先级**调整，**不是排除**。非画像记忆照常生成，不禁止、不删减。
- 以前 LLM 容易先去记知识点，现在要**先记画像**。

## 2. 不能照抄：母项目 vs 本插件的硬差异

母项目方法论可以学，但**格式/契约绝不能照搬**。下表是必须保持的差异：

| 维度 | 母项目（大型 agent） | 本插件（群聊）必须保留 |
|---|---|---|
| 顶层 JSON | `title/summary/openLoops/usefulMemoryIds/memoryActions` | `soul_state_code` + `feedback_data{useful_memory_ids, memory_actions}` |
| 命名风格 | camelCase | snake_case |
| 动作名 | `create/update/merge` | `create/merge/updata`（拼写为 updata） |
| 类型枚举 | `knowledge/skill/emotion/event` | `knowledge/skill/emotional/event`（emotional） |
| 用户标识 | `user-persona`（单用户） | 真实 QQ 号（纯数字 > 5 位，多群友并存） |
| 昵称/ID 获取 | 后置元信息 `[昵称] \| user_id=...` | 对话内联 `[群友: 昵称（QQ号）]（时间）:` |
| 画像属性 | 6 种（含「用户要求」） | 代码 `PROFILE_ATTRIBUTE_TAGS` 仅 5 种 |
| 任务交接 | summary / openLoops | 无（本插件只产记忆 + 灵魂码，不做任务交接） |

这些差异由下游代码强约束（`memory_id_resolver.py`、`user_profile.py`、
`json_parser.py`、`deepmind.py`），改提示词不得违反。

## 3. 下放清单（学方法论，不学格式）

✅ **下放**：
1. **办事指南式写法**：分步骤、可操作，替代当前零散罗列。
2. **归属判断**：先判断一条信息归「AI 自身经验」还是「用户画像」。
   - 本插件群聊语境下，「AI 自身经验」对应**与用户的共同回忆/体会**。
3. **五问筛选法**：无依据 / 临时状态 / 低价值噪音 / 高敏 / 贬损推测 → 不记。
4. **核心原则**：「宁可多记一条后续修正，也不要漏掉可能有用的信息」。
5. **画像为空 = 最高优先级**：和用户聊了一整轮却输出空 memoryActions 是不可接受的遗漏；
   画像为空时必须从对话中提炼至少一条画像（前提：稳定、有据、非敏感、非临时）。
6. **自然流露/稳定推断即可记**：不要求用户亲口说「我是 XX」，
   反复聊某游戏机制即可推断为深度玩家等。
7. **judgment / reasoning / tags 三段写法的具体要求**（陈述句独立可懂、
   reasoning 只写依据、tags 为检索锚点）。

✅ **额外引入（用户拍板）**：
8. **「」竖引号约定**：所有 JSON 字符串内容内部禁止出现未转义的英文双引号 `"`；
   需引用用户原话/字段名/命令/文件名时一律用「」，嵌套用『』。
   原因：实测中 JSON 字符串内的英文双引号经常导致反序列化失败。

❌ **不下放**（结构/契约/群聊约束）：
- 顶层 title/summary/openLoops、camelCase、update/emotion 命名、user-persona。
- 母项目「后置元信息取昵称」规则 → 改写为贴合本插件群聊内联格式。
- 第 6 个画像属性「用户要求」（本轮不引入，详见第 5 节）。

## 5. 决策已定（用户确认）

- ①「AI 自身经验」在群聊语境统一改写为「与用户的共同回忆/体会」。✓
- ②「画像优先」做**软倾向** + 「画像为空必须补一条」的硬底线。✓
- ③ **引入「」竖引号约定**：解决 JSON 字符串内英文双引号反序列化不兼容问题。✓

本轮仍不引入「用户要求」画像属性：母项目有 6 种，本插件代码
`PROFILE_ATTRIBUTE_TAGS` 仅 5 种，写入会导致下游识别不到画像。「以后都用中文」
这类先归「事实属性」；正式新增需下一轮「提示词 + 代码同步」改造。

> 「」竖引号只是**记忆文本写法约束**，不改任何字段名/JSON 结构，下游解析无感知，
> 因此本轮可安全引入。
