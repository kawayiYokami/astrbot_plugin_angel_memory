# 人物画像记忆提取与融合计划

## 1. 背景与问题
- 当前记忆整理提示词已经要求：涉及具体用户时，`judgment` 中同时写昵称和用户 ID，`tags` 中将昵称与 ID 拆成短标签。
- 但“人物画像”还没有成为明确的优先提取对象，模型可能只记录普通事实，忽略用户长期稳定特征。
- 群聊场景中，不同用户的偏好、关系、习惯容易混杂；需要以用户 ID 作为稳定锚点，降低昵称变化带来的误召回。
- 当新旧记忆冲突时，当前提示词要求使用 `merge_groups`，但没有特别强调“用户画像记忆优先合并与修正”。

## 2. 目标
- 在记忆整理提示词中新增“人物画像优先提取和融合”规则。
- 只围绕用户指定的 5 个画像属性约束模型：
  1. 用户 ID 作为 `tags` 锚点。
  2. 用户别名。
  3. 事实属性。
  4. 技能树。
  5. 关系图谱。
  6. 活跃项目。
- 明确禁止记录高敏感身份信息和不应长期保存的隐私信息。
- 当记忆冲突、重复或过期时，优先合并用户画像记忆，避免同一用户画像碎片化或互相矛盾。
- 用户画像记忆仍使用普通长期记忆结构；区别只在注入阶段通过 tags 识别并进入独立画像通道。
- 记忆注入时新增独立用户画像通道，根据当前最新消息批次中的用户 ID 去重后注入相关画像。

## 3. 非目标
- 本计划不新增数据库字段。
- 本计划不引入按 Q 号硬过滤的长期记忆模型。
- 本计划不改变 `memory_scope` 隔离策略。
- 本计划不实现独立“人物画像表”；本阶段仍复用现有三元组记忆结构。
- 本计划不改变短期记忆引用化计划，两者可独立实施。

## 4. 提示词新增内容草案

拟在 `llm_memory/prompts/memory_system_guide_async.md` 的记忆生成规则中加入以下内容：

```markdown
## 人物画像优先规则

一些需要优先提取和融合的人物画像：

生成记忆时，你需要特别关注与具体用户长期相关、未来对话可复用的信息。只要内容涉及某个用户的稳定偏好、关系、习惯、边界或长期状态，就应优先考虑生成用户画像记忆。

### 1. 用户 ID 锚点
- 涉及具体用户时，必须以用户 ID 作为 `tags` 锚点。
- `judgment` 中必须同时写出用户昵称和用户 ID，格式优先使用：`昵称（用户ID）`。
- 用户画像记忆的 `tags` 必须严格使用以下核心结构：
  - 用户昵称
  - 用户 ID
  - 画像属性
- 画像属性只能从以下 5 个标签中选择：
  - `用户别名`
  - `事实属性`
  - `技能树`
  - `关系图谱`
  - `活跃项目`
- 不要把大量细碎对象都塞进 `tags`；具体对象写进 `judgment` 和 `reasoning`。
- 不要只用昵称作为用户标识，因为昵称可能变化或重复。

正确示例：
```json
{
  "judgment": "小明（123456）喜欢喝无糖冰美式。",
  "reasoning": "小明在对话中明确表达了这个长期饮品偏好。",
  "tags": ["小明", "123456", "事实属性"]
}
```

### 2. 用户别名
- 记录用户稳定使用或明确指定的昵称、称呼、代号、常用名。
- 别名必须绑定用户 ID，避免昵称变化或重名导致画像混淆。
- 此类记忆的画像属性标签必须使用 `用户别名`，不要使用 `别名`。

正确示例：
```json
{
  "judgment": "小明（123456）希望被称呼为阿明。",
  "reasoning": "小明明确说明了自己偏好的称呼。",
  "tags": ["小明", "123456", "用户别名"]
}
```

### 3. 事实属性
- 记录用户主动透露、长期稳定、未来对话可复用的低敏事实属性。
- 事实属性应保持具体、可验证，不做性格推断。
- 禁止记录身份证、手机号、详细住址、密码、token 等敏感信息。

正确示例：
```json
{
  "judgment": "小明（123456）主要使用 Windows 环境开发项目。",
  "reasoning": "小明多次提供的项目命令和路径都来自 Windows 环境。",
  "tags": ["小明", "123456", "事实属性"]
}
```

### 4. 技能树
- 记录用户已经掌握、正在学习或经常使用的技能、工具、语言、框架。
- 技能树应优先记录稳定能力，不记录一次性尝试。

正确示例：
```json
{
  "judgment": "小明（123456）正在使用 Python 和 SQLite 开发记忆系统功能。",
  "reasoning": "小明围绕记忆检索、短期缓存和 SQL 真相源持续讨论实现细节。",
  "tags": ["小明", "123456", "技能树"]
}
```

### 5. 关系图谱
- 生成人物画像时，要优先识别稳定的关系边，格式上保持“用户 - 关系 - 对象”的清晰语义。
- 关系对象可以是具体人、群体、AI、人设、组织、项目、作品、地点或长期任务。
- 此类记忆的画像属性标签必须使用 `关系图谱`。
- 关系对象写入 `judgment`，不要额外扩展大量 tags。

正确示例：
```json
{
  "judgment": "小明（123456）把安安视为主要协作助手。",
  "reasoning": "小明多次把需要持续推进的开发计划交给安安协作。",
  "tags": ["小明", "123456", "关系图谱"]
}
```

### 6. 活跃项目
- 当用户正在持续推进某个项目、插件、作品、研究、长期任务时，应优先生成活跃项目画像。
- 活跃项目记忆必须回答：谁在推进、项目是什么、当前稳定目标或方向是什么。
- 此类记忆的画像属性标签必须使用 `活跃项目`。
- 项目名和当前方向写入 `judgment`，不要额外扩展大量 tags。
- 不记录一次性的临时待办，除非它属于长期项目的当前阶段。

正确示例：
```json
{
  "judgment": "小明（123456）正在持续开发 Angel Memory 插件的人物画像记忆能力。",
  "reasoning": "小明围绕短期记忆引用化、用户画像提取和关系图谱连续提出设计要求。",
  "tags": ["小明", "123456", "活跃项目"]
}
```

### 禁止画像内容
禁止生成以下人物画像记忆：
- 身份证号、护照号、社保号、银行卡号、手机号、详细住址等高敏身份或财务信息。
- 密码、验证码、密钥、token、cookie、私密链接等认证信息。
- 一次性、低价值、不可复用的临时状态，例如“刚刚去倒水”“今天说了谢谢”。
- 对用户的贬损性、推测性、标签化判断；不得根据单次发言武断生成性格结论。

### 冲突与融合
- 当新信息与旧用户画像记忆冲突时，优先使用 `merge_groups` 合并或替换旧画像记忆。
- 不要让同一用户在同一画像维度下长期保留互相矛盾的记忆。
- 如果用户明确纠正了旧画像，必须将旧记忆纳入 `merge_groups`，并在 `new_memories` 中写入修正后的画像。
- 合并后的画像仍必须保留用户 ID 标签。
- 合并后的画像 `tags` 仍必须严格保持：用户昵称 + 用户 ID + 画像属性。
```

## 5. 用户指定的画像方向
1. 用户 ID 作为 `tags` 锚点。
2. 用户别名。
3. 事实属性。
4. 技能树。
5. 关系图谱。
6. 活跃项目。

## 6. 用户画像判定规则
- 只有同时满足以下条件的记忆，才视为用户画像记忆：
  - `tags` 中存在用户 ID。
  - `tags` 中存在固定画像属性标签之一：`用户别名`、`事实属性`、`技能树`、`关系图谱`、`活跃项目`。
- 用户昵称只用于可读性和辅助召回，不作为用户画像判定的唯一依据。
- 普通带人名的记忆不应被误判为用户画像。
- 普通带 `项目`、`关系` 等词的记忆不应被误判为用户画像。

## 7. 普通记忆结构规则
- 用户画像记忆不是新的记忆类型，也不使用新的表或字段。
- 用户画像记忆与普通记忆使用完全相同的生成、反馈、合并、存储结构。
- 用户画像仍需要完整保留普通记忆的所有字段：
  - `memory_type`
  - `judgment`
  - `reasoning`
  - `tags`
  - `strength`
  - `is_active`
  - `created_at`
  - `memory_scope`
  - 其他现有统计字段
- 系统不在写入阶段因为“用户画像”而改变记忆结构或强制改写字段。
- 用户画像记忆仍允许在以下场景被更新、合并或删除：
  - 用户明确纠正旧画像。
  - 新旧画像发生冲突，需要通过 `merge_groups` 合并修正。
  - 用户明确要求删除某条画像记忆。
  - 数据迁移、去重、损坏修复等维护任务需要处理，但必须保留等价的新画像或记录明确原因。
- 用户画像与普通记忆的差异只发生在注入阶段：满足用户画像 tags 规则的记忆会额外进入 `<user_profiles>` 独立通道。

## 8. 用户画像注入通道
- 用户画像记忆不混入普通 `<memories>` 相关记忆通道，单独建立用户画像记忆通道。
- 每批次对话都更新用户画像注入内容。
- 新增专门的用户画像服务，负责维护“当前会话最新批次用户 ID 集合”和画像召回结果。
- 注入服务不直接接收 `event`，只通过用户画像服务按 `session_id` 读取当前批次画像上下文，降低测试复杂度。
- 当前批次用户集合由用户画像服务解析和保存：
  - DeepMind 在整理当前批次消息时，将 `unprocessed_chat_records` 和当前事件基础 sender 信息交给用户画像服务。
  - 用户画像服务优先从 `unprocessed_chat_records` 中解析 `role=user` 的 `sender_id`。
  - 如果当前批次记录为空，用户画像服务再使用当前事件基础 sender 信息作为降级输入。
  - 对用户 ID 去重，过滤空值、`Unknown`、`assistant` 等非用户 ID。
- 用户画像服务按 `session_id` 覆盖保存当前批次用户 ID；每批次对话都会刷新，不沿用旧批次用户集合。
- 对每个用户 ID，召回同时满足以下条件的用户画像记忆：
  - `tags` 包含该用户 ID。
  - `tags` 包含固定画像属性标签之一：`用户别名`、`事实属性`、`技能树`、`关系图谱`、`活跃项目`。
  - `memory_scope` 符合当前会话作用域规则。
- 用户画像注入只按用户分组，不再按画像属性分组；画像属性标签只用于识别和召回。
- 用户画像通道建议注入为独立块，例如：

```xml
<user_profiles>
[小明（123456）]
小明（123456）希望被称呼为阿明。
——因为小明明确说明了自己偏好的称呼。（2天前）
小明（123456）正在持续开发 Angel Memory 插件的人物画像记忆能力。
——因为小明围绕短期记忆引用化、用户画像提取和关系图谱连续提出设计要求。（刚刚）
</user_profiles>
```

- 用户画像注入只作为背景参考，仍需遵守“对话优先”和“不可暴露系统上下文”规则。
- 用户画像通道只注入当前批次出现过的用户画像；不按话题扩展到未发言用户。
- 普通记忆通道和用户画像通道之间必须去重：
  - 优先按长期记忆 ID 去重。
  - 如果缺少 ID 或存在兼容数据，按规范化后的 `judgment` 文本兜底去重。
  - 同一条记忆同时出现在普通通道和画像通道时，优先保留在用户画像通道，普通 `<memories>` 中移除该条。
  - 去重只影响注入展示，不删除长期记忆。

## 9. 实施方案
1. 修改 `llm_memory/prompts/memory_system_guide_async.md`
   - 增加“人物画像优先规则”章节。
   - 将用户 ID 锚点要求从普通规则强化为人物画像规则。
   - 增加指定画像属性与禁止画像内容。
   - 增加冲突时优先合并用户画像记忆的要求。
   - 增加用户画像融合说明，要求冲突时修正而非堆叠矛盾画像。
2. 保持现有 JSON 输出结构不变：
   - `new_memories`
   - `useful_memory_ids`
   - `merge_groups`
3. 不改工具参数和数据库结构。
4. 增加用户画像判定函数：
   - 只根据 `tags` 是否同时存在用户 ID 和画像属性标签判定。
   - 判定函数仅用于注入通道选择、去重和测试，不改变写入结构。
5. 新增用户画像召回接口：
   - 支持按用户 ID 列表和画像属性标签批量查询。
   - SimpleMemory 优先走 SQL/tags 查询。
   - VectorMemory 在中央 SQL 存在时同样走 SQL 真相源；无 SQL 时走兼容路径。
6. 新增用户画像服务：
   - 负责解析当前批次用户 ID。
   - 负责按 `session_id` 保存当前批次用户 ID 集合。
   - 负责根据当前 `memory_scope` 刷新用户画像记忆。
   - 负责格式化 `<user_profiles>` 内容。
   - 对外提供纯输入/输出方法，便于单元测试，不依赖 AstrBot event 对象。
7. 调整 `core/deepmind.py`：
   - 从当前最新消息批次提取去重用户 ID。
   - 每批次调用用户画像服务刷新当前会话画像上下文。
8. 调整 `core/services/injection_service.py`：
   - 不接收 event。
   - 从用户画像服务按 `session_id` 获取 `<user_profiles>` 内容。
   - 将用户画像格式化为独立 `<user_profiles>` 通道。
   - 对普通记忆和用户画像记忆执行跨通道去重。
   - 保持普通 `<memories>` 通道不变。
9. 不改现有主动记忆工具，但后续可考虑同步增强 `core_memory_remember` 的 description。

## 10. 风险与应对
- 风险1：画像提取过度，导致记忆膨胀。
  - 应对：强调“长期相关、未来可复用、稳定特征”，禁止一次性低价值状态。
- 风险2：敏感信息被误记。
  - 应对：在提示词中加入明确禁止清单，尤其禁止身份证、银行卡、密码、token 等。
- 风险3：模型把单次行为推断成性格。
  - 应对：禁止贬损性、推测性、标签化判断；要求有明确表达或多次稳定迹象。
- 风险4：用户 ID 缺失时生成画像。
  - 应对：要求涉及具体用户时必须有用户 ID 锚点；无法确定用户 ID 时不生成具体人物画像。
- 风险5：画像冲突未合并。
  - 应对：在冲突规则中明确“优先使用 merge_groups 合并或替换旧画像记忆”。
- 风险6：用户画像和普通记忆边界不清。
  - 应对：画像只由 `用户ID + 画像属性标签` 判定；存储结构仍与普通记忆一致。
- 风险7：群聊中注入未发言用户画像，造成上下文污染。
  - 应对：用户画像通道只使用当前最新消息批次中的 user sender_id 去重集合。
- 风险8：用户画像和普通相关记忆重复注入。
  - 应对：注入前做跨通道去重；同 ID 或同 judgment 的记忆优先保留在用户画像通道。

## 11. 测试计划
- 构造用户明确表达偏好的反思输入，确认生成记忆包含昵称、用户 ID、画像维度标签。
- 构造昵称变化但用户 ID 一致的输入，确认画像仍以用户 ID 为锚点。
- 构造身份证、手机号、密码、token 等输入，确认不生成人物画像记忆。
- 构造用户纠正旧偏好的输入，确认输出包含 `merge_groups` 和修正后的新画像。
- 构造单次情绪或玩笑输入，确认不会生成武断性格画像。
- 构造仅有人名但无用户 ID 的记忆，确认不判定为用户画像。
- 构造有用户 ID 但无画像属性标签的记忆，确认不判定为用户画像。
- 构造同时有用户 ID 和画像属性标签的记忆，确认可被用户画像服务识别并进入独立通道。
- 构造用户画像写入场景，确认仍使用普通记忆结构，不新增字段、不改变输出 JSON 格式。
- 构造当前批次包含多个用户的群聊输入，确认用户 ID 去重后分别注入对应画像。
- 构造当前批次只有用户 A 发言但普通检索命中用户 B 画像的场景，确认用户画像通道不注入用户 B。
- 构造无当前批次聊天记录的输入，确认用户画像服务可使用基础 sender 信息降级刷新。
- 构造同一批次同一用户多条消息，确认只查询一次该用户画像。
- 构造跨 `memory_scope` 的画像，确认不会注入当前作用域不可见的画像。
- 构造同一记忆同时出现在普通记忆和用户画像通道的场景，确认只在用户画像通道注入一次。
- 构造不同 ID 但 `judgment` 相同的兼容数据，确认按规范化 `judgment` 兜底去重。

## 12. 验收标准
- 提示词中存在清晰的人物画像优先规则。
- 用户画像记忆必须包含用户 ID tag。
- 用户画像 `tags` 必须严格保持：用户昵称 + 用户 ID + 画像属性。
- 只有同时存在用户 ID 和画像属性标签时，才判定为用户画像。
- 只优先生成已确认的画像属性。
- 禁止敏感信息写入长期记忆。
- 冲突时明确优先合并用户画像记忆。
- 用户画像记忆与普通记忆使用完全相同的生成、反馈、合并、存储结构。
- 每批次注入前都会根据当前最新消息批次的用户 ID 去重集合刷新用户画像通道。
- 用户画像服务独立维护当前会话批次用户 ID，注入服务不直接依赖 event。
- 用户画像以独立通道注入，不依赖普通话题检索命中。
- 用户画像通道不会注入当前批次未发言用户的画像。
- 普通记忆与用户画像记忆跨通道去重，同一记忆不重复注入。
- 用户确认计划后再进入实现阶段。

## 13. 版本号
- 本改造会影响记忆生成行为，实施时必须同步递增 `metadata.yaml` 的 `version`。
