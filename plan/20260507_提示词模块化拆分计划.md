# 提示词模块化拆分计划

## 1. 背景与问题
- 当前反思提示词只有一个文件：`llm_memory/prompts/memory_system_guide_async.md`。
- 加载方式也是单文件整读：
  - `core/utils/small_model_prompt_builder.py`
  - `llm_memory/service/cognitive_service.py`
- 这会导致每次修改任何一块规则时，都要编辑整份大提示词，极易出现：
  - 改了 JSON 协议，误删用户画像规则
  - 改了动作说明，误伤禁止项
  - 改了示例，污染其他章节

## 2. 目标
- 将当前“大一统提示词”拆分成稳定的多个模块文件。
- 保留一个统一的最终加载入口，但不再手写维护整份最终文案。
- 让“协议结构”“用户画像”“动作语义”“禁止项”“示例”等内容彼此隔离，降低误改概率。

## 3. 非目标
- 本次不修改记忆回想规则。
- 本次不改变 `memory_actions` 语义。
- 本次不改动数据库结构。
- 本次不改变小模型调用时机。

## 4. 拆分设计

建议将提示词拆成以下模块：

### 4.1 核心骨架
- 文件：`llm_memory/prompts/sections/00_intro.md`
- 内容：
  - 角色定义
  - “只输出 JSON”
  - 总体任务说明

### 4.2 输出协议
- 文件：`llm_memory/prompts/sections/10_output_schema.md`
- 内容：
  - 完整 JSON 示例
  - `soul_state_code`
  - `feedback_data.useful_memory_ids`
  - `feedback_data.memory_actions`

### 4.3 动作语义
- 文件：`llm_memory/prompts/sections/20_actions.md`
- 内容：
  - `create`
  - `merge`
  - `updata`
  - 三者区别
  - `merge` 多删一增 / `updata` 单删一增

### 4.4 通用记忆字段规则
- 文件：`llm_memory/prompts/sections/30_memory_fields.md`
- 内容：
  - `type`
  - `judgment`
  - `reasoning`
  - `tags`
  - tags 正误示例

### 4.5 用户画像规则
- 文件：`llm_memory/prompts/sections/40_user_profiles.md`
- 内容：
  - 用户 ID 锚点
  - 五类画像：
    - `用户别名`
    - `事实属性`
    - `技能树`
    - `关系图谱`
    - `活跃项目`
  - 禁止画像内容
  - 画像冲突与修正
  - 用户画像判定规则

### 4.6 通用生成与禁止规则
- 文件：`llm_memory/prompts/sections/50_generation_rules.md`
- 内容：
  - MUST
  - MUST NOT

### 4.7 专门示例
- 文件：`llm_memory/prompts/sections/60_examples.md`
- 内容：
  - 纯新增
  - 相似记忆合并
  - 冲突记忆 `updata`
  - 错误示例
  - 用户画像示例

### 4.8 最终自检与收尾
- 文件：`llm_memory/prompts/sections/70_checklist.md`
- 内容：
  - 生成流程
  - 自检清单
  - 最终输出要求

## 5. 组装方案

### 5.1 新增提示词构建器
- 新建：`llm_memory/prompts/prompt_assembler.py`
- 职责：
  - 按固定顺序读取 `sections/*.md`
  - 统一拼接成完整提示词
  - 返回给 `PathManager` / `SmallModelPromptBuilder` / `CognitiveService`

### 5.2 保留单一加载入口
- 不再让业务层直接逐个文件读取 section。
- 业务层仍然只调用一个统一接口，例如：
  - `PathManager.get_prompt_path()` 改为 `PathManager.get_prompt_dir()` 或保留原接口并改成读取组装结果
  - `SmallModelPromptBuilder` 和 `CognitiveService` 统一调用 assembler

### 5.3 最终文件策略
可选两种：

方案 A：运行时拼接
- 每次读取 sections 并现场拼接
- 优点：改动一个 section 立刻生效
- 缺点：读取逻辑稍多

方案 B：源码中保留 section，运行时组装字符串
- 不额外生成最终 markdown 文件
- 优点：单一真相源最清晰
- 缺点：无法直接打开一个最终静态文件检查

当前更推荐方案 B。

## 6. 兼容策略
- 旧文件 `memory_system_guide_async.md` 不再作为人工维护主文件。
- 可以保留为薄入口：
  - 仅写说明，提示不要直接编辑
  - 或彻底停用，改为 assembler 统一生成字符串

推荐：直接停用“手改主文件”模式，避免再次发生覆盖性误改。

## 7. 实施步骤
1. 创建 `sections/` 目录并拆分现有提示词内容。
2. 新增 `prompt_assembler.py`。
3. 修改 `SmallModelPromptBuilder` 的反思提示词加载逻辑。
4. 修改 `CognitiveService.get_prompt()` 的加载逻辑。
5. 清理旧的单文件直读逻辑。
6. 补充测试，确保组装顺序稳定、关键章节不丢失。
7. 更新 `metadata.yaml` 版本号。

## 8. 验收标准
- 提示词各大规则分文件维护，不再是单一巨型文件。
- 修改动作协议时，不会再误删用户画像章节。
- 修改用户画像规则时，不会影响动作协议章节。
- `SmallModelPromptBuilder` 与 `CognitiveService` 都从统一 assembler 获取完整提示词。
- 测试能验证最终提示词中至少包含：
  - `memory_actions`
  - `updata`
  - `人物画像优先规则`
  - `禁止画像内容`

## 9. 风险与注意事项
- 若同时保留“可手改主文件”和“section 组装”两套真相源，后续仍会混乱。
- section 顺序必须固定，否则提示词语义会漂移。
- JSON 示例、动作规则、用户画像规则必须分离，避免再次耦合。
