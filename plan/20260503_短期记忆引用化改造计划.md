# 短期记忆引用化改造计划

## 1. 背景与问题
- 当前会话工作记忆缓存使用 `MemoryItem` 保存召回结果快照，字段包含 `id`、`judgment`、`reasoning`、`tags`、`strength`、`created_at` 等。
- 这会让短期仓在语义上接近“第二份小型记忆库”，存在以下问题：
  - 长期记忆被修正、合并、删除后，短期仓可能继续持有旧文本。
  - 注入给 LLM 的内容不一定来自当前长期记忆真相源。
  - 合并/删除后的旧记忆如果仍在短期仓中，可能把已经失效的认知重新注入上下文。
- 期望将短期仓语义收敛为“当前会话持有的一组长期记忆引用”，事实内容只在注入前从长期库按 ID 读取。

## 2. 目标
- 短期仓只缓存记忆引用与调度元数据，不缓存事实字段。
- 注入前按短期仓中的记忆 ID 批量回查长期记忆真相源。
- 回查成功时，使用最新的 `judgment + reasoning + 相对时间` 格式化注入。
- 回查不到的记忆视为已失效，直接从短期仓移除，不使用快照降级。
- 保留现有按类型容量裁剪、昵称/印象优先、用户相关记忆容量均衡等短期仓策略。

## 3. 非目标
- 本阶段不改变长期记忆的存储模型。
- 本阶段不新增按 Q 号硬绑定的长期记忆过滤字段。
- 本阶段不改 `memory_scope` 隔离策略。
- 本阶段不改变检索排序、BM25、向量索引、rerank 等长期记忆召回逻辑。
- 本阶段不引入短期仓持久化；短期仓仍保持内存级会话工作缓存。

## 4. 核心原则
- 短期记忆只缓存引用，不缓存事实。
- 长期记忆库是唯一事实源。
- 查不到就消失，不使用 `fallback_snapshot`。
- 相对时间在每次注入前动态生成。
- 短期仓的额外字段只能用于调度、裁剪和分组，不能作为事实内容注入。

## 5. 数据结构方案

### 5.1 新短期引用结构
将当前 `MemoryItem` 调整为引用型结构，或新增 `SessionMemoryRef` 后逐步替换：

```python
@dataclass
class SessionMemoryRef:
    id: str
    memory_type: str
    priority_points: int = 3
    added_at: float = 0.0
    user_id: str = ""
```

字段说明：
- `id`：长期记忆 ID，注入前按此回查。
- `memory_type`：用于按类型容量裁剪，避免裁剪时频繁查库。
- `priority_points`：用于短期仓排序与淘汰。
- `added_at`：记忆进入短期仓的时间，用于稳定排序和旧项淘汰。
- `user_id`：可选调度元数据，用于用户相关记忆容量均衡；不作为事实来源。

### 5.2 不再缓存的字段
短期仓不再保存以下事实字段：
- `judgment`
- `reasoning`
- `tags`
- `strength`
- `created_at`

这些字段在注入前统一从长期记忆库按 ID 获取。

## 6. 流程设计

### 6.1 加入短期仓
长期记忆召回后：
1. 对每条 `BaseMemory` 提取 `id`。
2. 提取 `memory_type` 作为裁剪元数据。
3. 从 `tags` 中提取可选 `user_id`，仅用于用户相关容量管理。
4. 写入短期仓引用列表。
5. 如果同一 ID 已存在，刷新其引用元数据与 `added_at`。

### 6.2 注入前刷新
构建 prompt 相关记忆时：
1. 从短期仓获取当前会话的记忆 ID 列表。
2. 调用记忆运行时或 SQL 管理器按 ID 批量读取最新 `BaseMemory`。
3. 对查不到的 ID：
   - 从短期仓删除；
   - 不注入；
   - 仅输出 DEBUG 聚合日志，避免高频 INFO 噪声。
4. 对查到的记忆：
   - 按短期仓引用顺序或现有类型分组顺序恢复排序；
   - 使用最新字段格式化为 `judgment + reasoning + 相对时间`；
   - 注入 LLM 上下文。

### 6.3 主动回忆工具
`core_memory_recall` 当前直接格式化本次检索结果，不经过会话短期仓，可保持现状。
如果后续主动回忆结果也进入短期仓，应同样只写入引用。

### 6.4 反馈与反思链路
反思完成后新增的长期记忆进入短期仓时，也只加入引用。
被合并或删除的旧记忆 ID 在下一次注入刷新时自动消失。

## 7. 关键改动文件
- `core/session_memory.py`
  - 将 `MemoryItem` 快照语义改为引用语义，或新增 `SessionMemoryRef`。
  - 调整新增、去重、容量裁剪、用户容量均衡逻辑，避免依赖事实字段。
  - 增加按失效 ID 清理短期仓的方法。
- `core/services/injection_service.py`
  - 注入前按短期仓 ID 批量回查最新长期记忆。
  - 查不到的 ID 从短期仓移除。
- `core/utils/memory_formatter.py`
  - 保留对最新 `BaseMemory`/引用回查结果的格式化能力。
  - 确保相对时间在格式化时实时计算。
- `core/memory_runtime/protocol.py`
  - 评估是否需要新增 `get_memories_by_ids(ids)` 统一接口。
- `core/memory_runtime/simple_memory_runtime.py`
  - 如新增统一接口，转发到 `MemorySqlManager.get_memories_by_ids`。
- `core/memory_runtime/vector_memory_runtime.py`
  - 如新增统一接口，优先通过中央 SQL 真相源读取；无 SQL 时走旧向量集合兼容路径。
- `llm_memory/components/memory_sql_manager.py`
  - 复用现有 `get_memories_by_ids` 能力；必要时补齐顺序保持和缺失 ID 返回规则。

## 8. 接口设计建议

### 8.1 统一运行时读取接口
推荐在 `MemoryRuntime` 增加：

```python
async def get_memories_by_ids(self, memory_ids: List[str]) -> List[BaseMemory]:
    ...
```

约定：
- 返回查到的记忆列表。
- 不为缺失 ID 构造占位对象。
- 调用方负责根据返回 ID 集合清理短期仓。
- 返回顺序尽量跟随输入 ID；若底层无法保证，调用方可自行按输入顺序重排。

### 8.2 短期仓清理接口
推荐在 `SessionMemoryManager` 增加：

```python
def remove_memory_ids_from_session(self, session_id: str, memory_ids: List[str]) -> None:
    ...
```

用于注入刷新时清除已失效引用。

## 9. 日志要求
- 用户可见行为变化使用中文日志。
- 高频清理不逐条 INFO。
- 推荐日志：
  - DEBUG：`[短期记忆] 清理失效引用 session=... count=...`
  - WARNING：长期库批量读取失败时输出异常信息。
- 不使用 `status=success/failed/skipped` 风格。

## 10. 测试计划
- 单元测试：短期仓添加记忆后只保存 ID 与调度元数据，不保存 `judgment/reasoning/tags`。
- 单元测试：注入前按 ID 回查最新内容，长期记忆文本更新后注入使用新文本。
- 单元测试：长期记忆 ID 查不到时，从短期仓移除并且不注入。
- 单元测试：按类型容量裁剪仍生效。
- 单元测试：用户相关记忆容量均衡仍可基于 `user_id` 元数据工作。
- 回归测试：`core_memory_recall` 主动回忆输出不受短期仓引用化影响。
- 回归测试：SimpleMemory 与 VectorMemory 两种运行时均可完成注入前回查。

## 11. 验收标准
- 短期仓不再保存事实字段快照。
- 被删除或合并掉的长期记忆不会继续从短期仓注入。
- 长期记忆内容更新后，下一次注入使用最新内容。
- 现有会话记忆容量管理行为保持一致或有明确等价替代。
- 自测通过后，等待用户在生产环境验证；用户明确反馈“通过”后再归档计划。

## 12. 版本号
- 本改造会影响运行行为，实施时必须同步递增 `metadata.yaml` 的 `version`。
