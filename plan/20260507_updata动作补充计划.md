# updata 动作补充计划

## 1. 背景
- 当前 `memory_actions` 只支持 `create` 和 `merge`。
- 其中“旧记忆与当前事实冲突，需要明确修正”的场景，虽然技术上可以复用单源 `merge`，但语义上不够直观。
- 需要补一个显式的 `updata` 动作，让反思模型在“纠正旧记忆”时有更直接的表达方式。

## 2. 目标
- 在协议层新增合法动作名 `updata`。
- `updata` 与 `merge` 走完全相同的执行路径，不新增底层分支实现。
- 在提示词中明确：
  - `create` 用于纯新增
  - `merge` 用于相似记忆合并/去重
  - `updata` 用于旧记忆与当前事实冲突后的修正替换

## 3. 非目标
- 不修改记忆回想规则。
- 不改动记忆检索、分组、阈值和衰减逻辑。
- 不新增数据库字段。

## 4. 协议设计

```json
{
  "action": "updata",
  "source_memory_ids": ["52"],
  "memory": {
    "type": "knowledge",
    "judgment": "用户要求所有记忆反馈统一采用 memory_actions 协议。",
    "reasoning": "用户本轮明确否定旧结构，并确认只保留新的动作式协议。",
    "tags": ["memory_actions", "反馈协议", "结构约束"]
  }
}
```

说明：
- `updata` 必须携带 `source_memory_ids` 和 `memory`。
- `updata` 只针对 1 条旧记忆，底层复用 `merge` 路径统一处理。
- 执行结果仍然是：创建动作里的最终新记忆，并删除旧记忆。

## 5. 实施范围
- `core/utils/memory_id_resolver.py`
  - 放开 `updata` 为合法动作。
- `core/deepmind.py`
  - 对 `updata` 一并执行短 ID 到长 ID 的 source 映射。
- `llm_memory/service/memory_manager.py`
  - 将 `updata` 纳入与 `merge` 相同的执行分支。
- `llm_memory/components/memory_sql_manager.py`
  - 将 `updata` 纳入 SQL 路径的同一分支。
- `llm_memory/prompts/memory_system_guide_async.md`
  - 重写动作说明与示例，强调 `updata` 的冲突修正语义。
- `tests/test_memory_feedback_actions.py`
  - 增加 `updata` 的解析与执行测试。
- `metadata.yaml`
  - 提升版本号。

## 6. 验收标准
- `memory_actions` 接受 `create` / `merge` / `updata` 三种动作。
- `updata` 输入后，实际写入与 `merge` 路径一致。
- 提示词明确区分三种动作的使用时机。
- 测试覆盖 `updata` 解析与替换路径。
