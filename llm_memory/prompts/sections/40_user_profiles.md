## 人物画像优先规则

生成记忆时，你需要特别关注与具体用户长期相关、未来对话可复用的信息。只要内容涉及某个用户的稳定偏好、关系、习惯、边界或长期状态，就应优先考虑生成用户画像记忆。

### 1. 用户 ID 锚点
- 涉及具体用户时，必须以用户 ID 作为 `tags` 锚点。
- `judgment` 中必须同时写出用户昵称和用户 ID，格式优先使用：`昵称（用户ID）`。
- 用户画像记忆的 `tags` 必须严格使用以下核心结构：
  - 用户昵称
  - 用户 ID
  - 画像属性
- 画像属性只能从以下 5 个标签中选择：
  - `用户别名`
  - `事实属性`
  - `技能树`
  - `关系图谱`
  - `活跃项目`
- 不要把大量细碎对象都塞进 `tags`；具体对象写进 `judgment` 和 `reasoning`。
- 不要只用昵称作为用户标识，因为昵称可能变化或重复。

### 2. 用户别名
- 记录用户稳定使用或明确指定的昵称、称呼、代号、常用名。
- 别名必须绑定用户 ID，避免昵称变化或重名导致画像混淆。
- 此类记忆的画像属性标签必须使用 `用户别名`，不要使用 `别名`。

### 3. 事实属性
- 记录用户主动透露、长期稳定、未来对话可复用的低敏事实属性。
- 事实属性应保持具体、可验证，不做性格推断。
- 禁止记录身份证、手机号、详细住址、密码、token 等敏感信息。

### 4. 技能树
- 记录用户已经掌握、正在学习或经常使用的技能、工具、语言、框架。
- 技能树应优先记录稳定能力，不记录一次性尝试。

### 5. 关系图谱
- 生成人物画像时，要优先识别稳定的关系边，格式上保持“用户 - 关系 - 对象”的清晰语义。
- 关系对象可以是具体人、群体、AI、人设、组织、项目、作品、地点或长期任务。
- 此类记忆的画像属性标签必须使用 `关系图谱`。
- 关系对象写入 `judgment`，不要额外扩展大量 `tags`。

### 6. 活跃项目
- 当用户正在持续推进某个项目、插件、作品、研究、长期任务时，应优先生成活跃项目画像。
- 活跃项目记忆必须回答：谁在推进、项目是什么、当前稳定目标或方向是什么。
- 此类记忆的画像属性标签必须使用 `活跃项目`。
- 项目名和当前方向写入 `judgment`，不要额外扩展大量 `tags`。
- 不记录一次性的临时待办，除非它属于长期项目的当前阶段。

### 禁止画像内容
- 禁止生成以下人物画像记忆：
  - 身份证号、护照号、社保号、银行卡号、手机号、详细住址等高敏身份或财务信息。
  - 密码、验证码、密钥、token、cookie、私密链接等认证信息。
  - 单次情绪、一次性口误、玩笑话、不可复用的临时状态。
  - 对用户的贬损性、武断性格判断、无依据推测。

### 画像冲突与修正
- 当新信息与旧用户画像冲突时，优先修正画像，不要长期并存互相矛盾的画像记忆。
- 发现同一用户在同一画像维度下有重复、过时或冲突记忆时：
  - 相似收敛优先使用 `merge`
  - 明确纠正旧事实优先使用 `updata`
- 若用户明确纠正旧画像，优先使用 `updata`，不能仅 `create` 一条新画像；且 `source_memory_ids` 只能写 1 条旧画像 ID。
- 合并或更新后的画像，`tags` 仍必须严格保持：用户昵称 + 用户 ID + 画像属性。

### 用户画像判定规则
- 只有同时满足以下条件的记忆，才视为用户画像记忆：
  - `tags` 中存在用户 ID
  - `tags` 中存在固定画像属性标签之一：`用户别名`、`事实属性`、`技能树`、`关系图谱`、`活跃项目`
- 用户昵称只用于可读性和辅助召回，不作为用户画像判定的唯一依据。
- 普通带人名的记忆不应被误判为用户画像。
- 普通带 `项目`、`关系` 等词的记忆不应被误判为用户画像。
